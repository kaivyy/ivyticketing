# Design — Phases 16–27 (Platform, Enterprise & Launch)

**Date:** 2026-07-07
**Scope:** Build phases 16–27 of `docs/masterplan.md` on top of the existing
phase 1–15 modular monolith. Autonomous batch execution with checkpoints every
3–4 phases.

## Guiding principles (from masterplan Final Priority Rule)

1. Correctness → 2. Stability → 3. Security → 4. Transparency → 5. Speed → 6. UI polish.

Every new module follows the **existing** architecture verbatim:
`handler → service → repository(interface) → db.Queries(sqlc)`, chi routes
registered in `server.go`, RBAC via `middleware.RequirePermission` + `rbac.Loader`,
audit via `audit.Logger`, migrations in goose single-file format, tests colocated
in `tests/`. New frontend under `apps/web` (Astro) reusing the three layouts and
local `ui/` components. No new frameworks.

## Batch 1 — Organizer value (Phases 16, 17, 18)

### Phase 16 — Reporting & Export
- **Module** `reporting`. Read-only aggregate queries over existing tables
  (orders, payments, tickets, queue, ballot, racepack, access/coupons).
- Report types: participant, sales, payment, coupon, queue, ballot, racepack, revenue.
- **Async export via worker**: `export_jobs` table (PENDING→PROCESSING→READY→FAILED),
  organizer requests export, worker generates CSV to storage, organizer downloads
  via signed/owned URL. Large exports never block the API (masterplan AC).
- Formats: CSV (worker), on-screen summary JSON. PDF summary = print-CSS on frontend (reuse Phase 7 invoice pattern).
- Permission: `report.view` (exists), `participant.export` (exists).
- Super admin: cross-org aggregate endpoint.
- **Frontend**: `org/[orgId]/reports/*` pages — report dashboard + export job list/download.

### Phase 17 — Super Admin Platform Billing
- **Module** `billing`. Subscription packages (Starter/Professional/Enterprise),
  per-org subscription, platform fee ledger, platform invoices.
- `subscription_packages`, `org_subscriptions`, `platform_invoices`, `platform_fee_ledger` tables.
- **Feature gating**: `PackageGate` service — resolves an org's active package and
  exposes `Can(feature)` (queue, ballot, racepack, whitelabel, api). Wired as a
  soft gate that other modules can consult; enforcement additive (default allow
  for existing orgs to avoid regressions, explicit deny only when a package is set).
- Platform fee computed on PAID orders (percentage of order total), recorded to ledger.
- Super-admin endpoints: manage packages, assign/upgrade org subscription, view revenue.
- Organizer endpoint: view own subscription + upgrade request.
- **Frontend**: `admin/*` (new platform-admin area) billing dashboard + `org/[orgId]/billing.astro`.

### Phase 18 — White Label & Custom Domain
- **Module** `whitelabel`. Per-org branding: logo, theme color, custom email sender,
  custom terms/footer, white-label toggle, custom domain + verification.
- `org_branding`, `custom_domains` (with DNS TXT verification token, status).
- Domain verification: issue TXT token, verify via lookup, mark VERIFIED. SSL is
  deployment-layer (documented, not app code — noted as external like Phase 9 WAF).
- Public catalog + participant pages read branding by org/domain to theme output.
- Gated behind Enterprise package (Phase 17 `PackageGate`).
- **Frontend**: `org/[orgId]/branding.astro`; public pages consume branding tokens.

**Checkpoint 1** after 18.

## Batch 2 — Reliability & operations (Phases 19, 20, 21)

### Phase 19 — Public Status & Incident System
- **Module** `status`. Component health (queue/payment/registration) +
  incident CRUD with updates timeline.
- `incidents`, `incident_updates`, `component_status` tables.
- Public endpoint `GET /api/v1/public/status` (no auth) — component states + active incidents + last-updated.
- Admin: create/update/resolve incident, set component status.
- Banner surfaces in queue + checkout pages (frontend reads status).
- **Frontend**: public `status.astro`; admin incident manager.

### Phase 20 — Observability & War Room Dashboard
- **Prometheus metrics** via `platform/metrics` + `promhttp` at `/metrics`.
  Counters/histograms: HTTP p95 (middleware), checkout success, payment success,
  queue active/release rate, webhook delay, scan rate, DB pool usage.
- Request ID already exists (Phase 1) → ensure propagated frontend→backend header.
- **War room dashboard**: authenticated super-admin page reading a JSON metrics
  snapshot endpoint (Grafana optional/external; ship an in-app snapshot so it works
  without extra infra). Prometheus scrape is the production path (documented).
- Sentry = env-configured DSN hook (optional, no-op if unset).
- **Frontend**: `admin/war-room.astro`.

### Phase 21 — Load Testing & Reliability Hardening
- k6 scripts under `tests/load/` for: checkout race, payment callback spike,
  queue 10k/50k/100k (extend existing `queue-war.js`), refresh storm.
- **Reliability**: verify graceful degradation paths already coded (Redis fail-open,
  gateway retry). Add a documented incident runbook `docs/RUNBOOK.md`.
- Correctness re-assertion tests: no oversold, no double payment, no queue reset
  (extend existing race tests). This phase is mostly test + docs, minimal new runtime code.

**Checkpoint 2** after 21.

## Batch 3 — Security, integration, post-race (Phases 22, 23, 24)

### Phase 22 — Security Hardening
- **Audit**: RBAC review pass, verify webhook signature + QR signature paths,
  confirm cross-event/cross-org isolation (defense-in-depth already partly done).
- CSRF: refresh-cookie flow uses SameSite=Lax + custom header already; document + add
  explicit check where state-changing + cookie-based.
- **Audit log immutability**: DB trigger blocking UPDATE/DELETE on `audit_logs`
  (masterplan known-gap from Phase 15 doc).
- PII: identify sensitive fields; ensure not logged (scrub middleware / log review).
- Run the repo's `security-review` skill over the diff.
- Output: `docs/SECURITY_HARDENING.md` with findings + fixes.

### Phase 23 — Enterprise API & Integration
- **Module** `enterprise`. Per-org API keys (hashed storage, prefix + secret),
  scoped read APIs: event, participant, order, payment-status, racepack, result-import.
- `api_keys` table (org-scoped, scopes, last-used, revoke). Rate limit per key
  (reuse `platform/ratelimit` with key subject).
- **Outbound webhooks**: `webhook_endpoints` + delivery with idempotency + signature
  (reuse HMAC pattern). Worker delivers, retries with backoff.
- API-key auth middleware distinct from JWT (Bearer `sk_...` → org context).
- Gated behind Enterprise package.
- **Frontend**: `org/[orgId]/api-keys.astro` + webhook endpoint management.

### Phase 24 — Result, Certificate & Timing
- **Module** `results`. Import results CSV (async via worker), rankings
  (overall/gender/age-group), DNF/DNS, certificate generation (HTML→print, reuse invoice pattern), public result page.
- `event_results` (ticket/bib → time, status, ranks), computed rank columns.
- Timing vendor API = pluggable interface stub (like storage/gateway), CSV import is the concrete path.
- Participant: view result + download certificate. Organizer: import + manage.
- **Frontend**: participant `results/[eventId].astro` + certificate; organizer results import page.

**Checkpoint 3** after 24.

- `post_event_reports` (generated snapshot), `improvement_backlog` (simple items list).
- **Frontend**: `org/[orgId]/events/[eventId]/post-event.astro`.

## Cross-cutting decisions

- **Migrations** start at 00054, goose single-file format, one logical change per file.
- **RBAC**: new permissions seeded via migration, granted to appropriate templates
  (Owner/Manager/Finance). Platform-admin-only features use `RequirePlatformAdmin`.
- **Feature gating (Phase 17)** is additive and default-allow for orgs without a
  subscription row, so existing phase 1–15 behavior never regresses.
- **Async work** (exports, result imports, webhook delivery) runs in the existing
  `cmd/worker` via new ticker jobs, following the ballot expirer / order expiration pattern.
- **Frontend admin area**: introduce `apps/web/src/pages/admin/*` + an `AdminLayout`
  for platform-admin (billing, war-room, incidents, abuse links).
- **Testing**: unit tests per service (table-driven, matching existing style),
  integration tests behind `integration` build tag against real Postgres, race tests
  where concurrency matters. `go build ./...` + `go test ./...` green before each commit.
- **Commits**: one commit per phase, `feat(phaseNN): ...`, matching existing history.

## Success criteria

Each phase's masterplan Acceptance Criteria met. Backend builds and all tests pass.
New UI reachable and styled consistently. No regression in phases 1–15 (existing
test suite stays green). Checkpoints reported to user at the batch boundaries.

## Out of scope (deferred, matching masterplan intent)

- Actual microservice split (25 is docs/seams only).
- Native mobile app, marketplace sponsor, hotel/travel, complex affiliate.
- External Grafana/Loki/Sentry infra provisioning (app exposes the hooks; infra is deployment-layer).
- Real SSL cert automation for custom domains (documented, deployment-layer).
